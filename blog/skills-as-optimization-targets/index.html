Separating Determinism from Reasoning

January 27, 2026

Agent workflows mix two types of work:

  • Reasoning — understanding context, making decisions, handling ambiguity
  • Determinism — running commands in sequence, updating files in predictable patterns

The reasoning is what LLMs are for. The determinism is just burning tokens.

The refactoring pattern

I've been using this prompt on my skills (markdown files that define agent workflows):

Review all skills. Look for opportunities to extract deterministic
            behavior into scripts or CLI commands that the agent can use instead
            of spending tokens doing it.

The LLM reads the workflow definitions and identifies patterns like:

  • Sequential commands with early exit (run checks, stop on first failure)
  • File updates across multiple locations (bump version in two files)
  • Structured text manipulation (insert entry into correct changelog section)
  • Conditional logic based on known state (different git diff based on branch)

For each pattern, it suggests a specific tool. Instead of the agent running four commands and interpreting results, a single check command with a structured exit code.

Why this works

The LLM is good at understanding intent and finding patterns — that's reasoning. Once it identifies a deterministic pattern, you extract it into code where it belongs.

The LLM finds the separation. Code enforces it.

This makes agents faster (fewer API calls), cheaper (fewer tokens), and more reliable (deterministic code vs probabilistic interpretation).

The feedback loop

  1. Write workflows as skills
  2. Use the LLM to audit for deterministic patterns
  3. Extract patterns into tooling
  4. Update skills to use the new tools
  5. Repeat

Each iteration moves more determinism out of the LLM and into code, leaving the agent to focus on what it's actually good at.
